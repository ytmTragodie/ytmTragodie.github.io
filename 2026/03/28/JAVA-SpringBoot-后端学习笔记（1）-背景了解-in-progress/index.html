Java Spring boot 后端笔记（1）–简介

Spring 框架简介

Spring 是一个轻量级的 Java 开发框架，旨在解决传统 Java EE 开发中代码臃肿、耦合度高、测试困难等问题。它最早由 Rod Johnson 在 2002 年提出，核心思想是 “让 Java 开发变得更简单、更优雅”。

Spring 提供了两大核心功能：

  • IoC（控制反转）容器：帮你管理对象的创建和依赖关系。
  • AOP（面向切面编程）：让你在不修改源代码的情况下，为程序添加额外功能（如日志、事务、权限等）。

这样做的问题：OrderService 和 PaymentService 高度耦合，难以替换、测试或扩展。

Spring 采用 IoC（控制反转） 思想：将对象的创建和依赖关系的维护交给容器来管理。只需要声明“我需要什么”，Spring 容器会在运行时自动把依赖的对象“注入”进来。

这种 DI（依赖注入） 机制让代码变得松散耦合，易于测试和修改。可以轻松地替换实现类，或者在单元测试中使用 Mock 对象。

2. AOP（面向切面编程）

AOP 允许将那些与业务无关但多个模块都会用到的功能（如日志、事务、权限校验）抽取出来，称为“切面”，然后在运行时动态地“织入”到业务方法中。

  • Spring 提供了基础能力（IoC、AOP、数据访问、MVC 等），但配置相对繁琐。你需要手动配置数据源、事务管理器、视图解析器等，还要管理各种依赖的版本。
  • Spring Boot 是在 Spring 之上构建的快速开发框架，它采用“约定优于配置”的思想，自动帮你完成这些配置，并提供起步依赖（Starter）来简化依赖管理。

Spring Boot

Spring Boot 是 Spring 框架的一个子项目，它并不是取代 Spring，而是在 Spring 的基础上做了一层“封装”和“自动化配置”，目的是让你以最小的代价、最少的配置快速创建一个独立运行的 Spring 应用。

可以理解为一个 “开箱即用的 Spring 应用脚手架”——不用再去手动配置一大堆 XML 或 Java Config，不用操心依赖版本兼容性，甚至不用自己部署 Web 服务器，因为 Spring Boot 已经内置了 Tomcat、Jetty 等服务器。

1.核心思想

Spring Boot 最核心的设计哲学是 “约定优于配置”（Convention over Configuration）。

传统 Spring 开发中，你需要写大量配置：数据源配置、事务管理、视图解析器、扫描包路径……稍不注意就会出错。Spring Boot 则根据“常规开发习惯”给出了一套默认的约定，如果你遵循这些约定，几乎不需要任何配置；如果你需要自定义，只需在配置文件里修改几个属性即可。

例如：

  • 默认扫描主启动类所在包及其子包下的组件（@SpringBootApplication 就包含了 @ComponentScan）
  • 默认的静态资源路径是 classpath:/static/、classpath:/public/
  • 默认的配置文件是 application.properties 或 application.yml

好处：只需要关注业务逻辑，框架帮你处理了底层繁琐的配置。

2.主要原理

Spring Boot 有两个核心机制：起步依赖（Starter） 和 自动配置（AutoConfiguration）。

Spring Boot 提供了一系列“场景启动器”，比如：

  • spring-boot-starter-web：包含 Spring MVC、内置 Tomcat、Jackson 等，用于开发 Web 应用
  • spring-boot-starter-data-jpa：包含 JPA、Hibernate、数据库连接池等
  • spring-boot-starter-test：包含 JUnit、Mockito 等测试依赖

只需要在 pom.xml 中添加一个 starter，所有相关的依赖就会自动加入，并且版本号经过充分测试，避免了依赖冲突。

当你的项目引入了某个 starter，Spring Boot 会根据 类路径下是否存在某个类 来自动配置相应的功能。

例如：

  • 类路径下有 H2Database 的类，就自动配置一个 H2 数据源
  • 类路径下有 Servlet、Spring MVC 相关类，就自动配置 DispatcherServlet 和嵌入式 Tomcat

背后的原理：Spring Boot 在启动时会加载 META-INF/spring.factories 文件中定义的所有自动配置类，这些配置类使用 @Conditional 系列注解（如 @ConditionalOnClass、@ConditionalOnMissingBean）来判断条件是否满足，从而决定是否生效。

入依赖 → 框架检测到依赖 → 自动帮你配置好对应的组件，你只需要写业务代码。

Spring Data JPA

Spring Data JPA 是 Spring Data 项目的一部分，它提供了一套基于 JPA 的 Repository 抽象。开发者只需定义继承自 JpaRepository 或 CrudRepository 的接口，Spring 就会在运行时自动生成实现类，并提供常见的 CRUD、分页、排序等操作。

对数据访问封装的原理

Spring Data JPA 的核心原理可以概括为：接口驱动 + 动态代理 + 方法名解析 + EntityManager 操作。

  • 开发者定义一个继承 Repository 标记接口的子接口（如 JpaRepository）。
  • 在 Spring 容器启动时，通过 @EnableJpaRepositories 或自动配置，Spring 会扫描指定包下的所有 Repository 接口。
  • 对每个接口，JpaRepositoryFactory 会创建一个 动态代理 实例。该代理在运行时拦截对接口方法的调用，并根据方法名或注解决定执行何种数据库操作。

方法名解析机制

当调用一个自定义方法（如 findByUsernameAndActiveTrue）时，代理会解析方法名：

  • 解析前缀（findBy、getBy、countBy 等）。
  • 提取属性路径（Username、Active）。
  • 组合成 JPQL 查询条件，例如：where username = ?1 and active = true。
  • 使用 EntityManager 创建 TypedQuery 并执行。

如果方法上标注了 @Query，则直接使用用户提供的 JPQL 或原生 SQL 语句，同样通过 EntityManager 执行。

底层依赖 EntityManager

所有 Repository 实现最终都依赖 JPA 的 EntityManager。Spring Data JPA 负责管理 EntityManager 的生命周期（通常是容器管理的 PersistenceContext），在代理方法内部调用 entityManager.persist()、entityManager.find()、entityManager.createQuery() 等 API 完成数据操作。

默认情况下，Repository 的 CRUD 方法都带有 @Transactional 注解（只读方法有 @Transactional(readOnly=true)），确保每个操作在事务上下文中执行。用户也可以扩展事务边界到 Service 层。

与其它数据访问方式的对比

方式 优点 缺点
Spring Data JPA 开发效率高，自动实现，减少样板代码 复杂查询可能需要 JPQL 或原生 SQL；过度依赖动态代理可能增加调试难度
MyBatis SQL 完全可控，适合复杂查询 需要手写 SQL 和映射，代码量较大
原生 JDBC 性能极致，完全控制 代码冗长，易出错，开发效率低

JPA Repository

JPA Repository 是 Spring Data JPA 提供的一个核心接口，它抽象了数据访问层的实现细节，让开发者能够以极简的方式执行常见的数据库操作。它几乎出现在所有使用 JPA（如 Hibernate）的 Spring Boot 后端项目中，是连接业务逻辑与关系数据库的关键桥梁。
